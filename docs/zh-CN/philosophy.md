# 设计理念

本文档解释了 Codingbuddy 的愿景、核心理念和设计原则。

## 愿景

**所有 AI 助手的统一编码规则来源**

如今的开发团队使用多种 AI 编程工具——Cursor、Claude Code、GitHub Copilot 等。每个工具都有自己独特的配置格式，导致：

- 多个配置文件中的规则重复
- 不同 AI 工具导致的编码标准不一致
- 规则更新时的维护负担

Codingbuddy 通过提供一个与任何 AI 助手兼容的统一规则系统来解决这个问题。

## 核心理念

### 1. 与 AI 无关的规则

规则只需编写一次，到处可用。无供应商锁定，核心规则中无工具特定语法。每个 AI 工具通过轻量级适配器适配通用格式。

### 2. 渐进式披露

从简单开始，需要时深入：

- **快速开始**：使用 `npx codingbuddy init` 2 分钟内运行
- **配置**：自定义技术栈、架构和规范
- **专家代理**：访问领域专家（安全、性能、可访问性）
- **完全自定义**：使用项目特定规则扩展

### 3. 约定优于配置

适用于大多数项目的合理默认值：

- PLAN → ACT → EVAL 工作流
- TDD 优先的开发方法
- 80%+ 测试覆盖率目标
- SOLID 原则和整洁代码

只需覆盖您需要更改的内容。

### 4. 社区驱动的标准

最佳实践来自真实世界的经验：

- 规则基于生产代码库中经过验证的模式
- 专家代理编码了从业者的领域专业知识
- 开源且欢迎贡献

## 设计原则

### 单一数据源

```
packages/rules/.ai-rules/           ← 权威来源
├── rules/           ← 核心规则（工作流、质量、项目）
├── agents/          ← 专家知识
└── adapters/        ← 工具特定的集成指南
```

所有 AI 工具配置都引用 `packages/rules/.ai-rules/`。更新一次，所有工具受益。

### 关注点分离

| 层 | 目的 | 格式 |
|---|------|------|
| **Rules** | 做什么（工作流、质量标准） | Markdown |
| **Agents** | 谁知道什么（专家知识） | JSON |
| **Adapters** | 如何集成（工具特定设置） | Markdown |

这种分离允许：

- 规则独立于工具支持演进
- 无需更改核心规则即可添加新代理
- 无需修改现有规则即可支持新工具

### 可扩展性优于复杂性

系统设计为可扩展而非可配置：

- 通过创建 JSON 文件添加新的专家代理
- 通过编写适配器指南支持新工具
- 无需修改核心规则即可包含项目特定上下文

简单的事情应该简单。复杂的事情应该可能。

## 工作流模型

Codingbuddy 为 AI 辅助开发引入了结构化的工作流：

```
PLAN → ACT → EVAL
```

### PLAN 模式（默认）

- 理解需求
- 设计实施方法
- 识别风险和边缘情况
- 不进行代码更改

### ACT 模式

- 执行计划
- 遵循 TDD：Red → Green → Refactor
- 进行增量的、经过测试的更改

### EVAL 模式

- 审查实施质量
- 识别改进点
- 建议重构机会

此工作流防止了 AI 助手在没有适当规划的情况下直接跳入代码的常见陷阱。

## Codingbuddy 不是什么

- **不是代码生成器**：它提供规则和上下文，而不是生成的代码
- **不是人类判断的替代品**：它增强而不是取代开发者的决策
- **不是一刀切的解决方案**：它被设计为按项目自定义
