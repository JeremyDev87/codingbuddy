# 設計思想

このドキュメントでは、Codingbuddyのビジョン、核となる信念、設計原則について説明します。

## ビジョン

**すべてのAIアシスタントのための統一されたコーディングルールソース**

今日の開発チームは、Cursor、Claude Code、GitHub Copilotなど、複数のAIコーディングツールを使用しています。各ツールには独自の設定形式があり、以下のような問題が発生します：

- 複数の設定ファイルでのルールの重複
- 使用するAIツールによって一貫性のないコーディング標準
- ルール更新時のメンテナンス負担

Codingbuddyは、あらゆるAIアシスタントと互換性のある統一ルールシステムを提供することで、この問題を解決します。

## 核となる信念

### 1. AI非依存のルール

ルールは一度書けばどこでも動作します。ベンダーロックインなし、コアルールにツール固有の構文なし。各AIツールは軽量アダプターを通じて共通フォーマットに適応します。

### 2. 段階的開示

シンプルに始めて、必要に応じて深掘り：

- **クイックスタート**: `npx codingbuddy init`で2分以内に実行
- **設定**: 技術スタック、アーキテクチャ、規約をカスタマイズ
- **スペシャリストエージェント**: ドメイン専門家にアクセス（セキュリティ、パフォーマンス、アクセシビリティ）
- **完全なカスタマイズ**: プロジェクト固有のルールで拡張

### 3. 設定より規約

ほとんどのプロジェクトで機能する合理的なデフォルト：

- PLAN → ACT → EVALワークフロー
- TDDファースト開発アプローチ
- 80%以上のテストカバレッジ目標
- SOLID原則とクリーンコード

変更が必要な部分だけをオーバーライドします。

### 4. コミュニティ主導の標準

ベストプラクティスは実際の経験から生まれます：

- ルールは本番コードベースの実証済みパターンに基づく
- スペシャリストエージェントは実務者のドメイン専門知識をエンコード
- オープンソースで貢献を歓迎

## 設計原則

### 単一ソース

```
packages/rules/.ai-rules/           ← 権威あるソース
├── rules/           ← コアルール（ワークフロー、品質、プロジェクト）
├── agents/          ← 専門知識
└── adapters/        ← ツール固有の統合ガイド
```

すべてのAIツール設定が`packages/rules/.ai-rules/`を参照します。一度更新すれば、すべてのツールに反映されます。

### 関心の分離

| レイヤー | 目的 | 形式 |
|----------|------|------|
| **Rules** | 何をするか（ワークフロー、品質標準） | Markdown |
| **Agents** | 誰が何を知っているか（専門知識） | JSON |
| **Adapters** | どう統合するか（ツール固有の設定） | Markdown |

この分離により：

- ルールはツールサポートとは独立して進化可能
- コアルールを変更せずに新しいエージェントを追加可能
- 既存ルールを修正せずに新しいツールをサポート可能

### 複雑性より拡張性

システムは設定ではなく拡張のために設計されています：

- JSONファイルを作成して新しいスペシャリストエージェントを追加
- アダプターガイドを書いて新しいツールをサポート
- コアルールを修正せずにプロジェクト固有のコンテキストを含める

シンプルなことはシンプルに。複雑なことは可能に。

## ワークフローモデル

CodingbuddyはAI支援開発のための構造化されたワークフローを導入します：

```
PLAN → ACT → EVAL
```

### PLANモード（デフォルト）

- 要件を理解する
- 実装アプローチを設計する
- リスクとエッジケースを特定する
- コード変更なし

### ACTモード

- 計画を実行する
- TDDに従う：Red → Green → Refactor
- 段階的でテストされた変更

### EVALモード

- 実装品質をレビューする
- 改善点を特定する
- リファクタリングの機会を提案する

このワークフローは、AIアシスタントが適切な計画なしにすぐにコードに飛び込むという一般的な落とし穴を防ぎます。

## Codingbuddyではないもの

- **コード生成器ではない**: 生成されたコードではなく、ルールとコンテキストを提供
- **人間の判断の代替ではない**: 開発者の意思決定を置き換えるのではなく補強
- **万能ソリューションではない**: プロジェクトごとにカスタマイズされるよう設計
